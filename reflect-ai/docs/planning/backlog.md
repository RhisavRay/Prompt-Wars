# Purpose

The Product Backlog contains approved work that has not yet been assigned to an active sprint.

Unlike the Ideas document, every item in this backlog has already been evaluated and accepted as valuable to Reflect.

The backlog serves as the bridge between long-term product vision and short-term sprint planning.

# Prioritization Framework

Features should be evaluated using the ICE framework.

Priority Score = (Impact × Confidence) ÷ Effort

Where:

- **Impact** – How much value does this create for users?
- **Confidence** – How confident are we in the solution?
- **Effort** – Estimated implementation complexity.

Features with higher scores should generally be prioritized.

---

# Before Moving A Feature Into A Sprint

Ask:

- Does this align with Reflect's philosophy?
- Does this improve the journaling experience?
- Is the architecture ready?
- Can it be completed within one sprint?
- Is this the highest-value feature available?

Only move features into a sprint when the answer to all of these questions is "Yes."

---

# Backlog Maintenance

Review the backlog regularly.

Items may:

- Move to a sprint.
- Change priority.
- Be merged with another feature.
- Be removed if they no longer align with Reflect.

A smaller, well-maintained backlog is more valuable than a large, outdated one.
